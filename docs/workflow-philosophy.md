# AI 协作工作流哲学

## 文档是基础设施

对 agent 而言，稳定的设计、决策和经验文档不是事后说明，而是跨会话恢复上下文、保持术语一致和避免重复犯错的基础设施。代码说明系统现在做什么；Design 和决策记录说明为什么这样做、哪些边界不能被随意改变。

## 架构先于实现

实现速度会放大设计质量。边界、组件职责、数据所有权、接口、实现机制、生产集成点和失败行为没有讲清楚时，快速编码只会更快地产生耦合与返工。Design 阶段的目标不是写得长，也不是把模板标题填满，而是通过调查和讨论把会改变实现方向的决定说清楚。

## 失败记录比成功叙述更有价值

成功方案可以从代码重建；失败尝试如果没有记录，后续 agent 会在相同假设上重复消耗。记录失败时应包含症状、证据、因果机制、为什么失败，以及什么条件变化后才值得重试。

## 会话延续必须系统化

不要依赖单个对话无限增长。项目应有稳定的启动阅读顺序、活跃 Design/Plan、会话结束记录和可复用 learnings。原始过程日志应定期压缩成决策或经验，而不是永久把全部历史塞进上下文。

## Slop 公式

`Slop = agent 速度 × 约束缺失 × 无验证 × 文档不足`

提高模型速度无法解决 slop；真正的杠杆是明确边界、选择能证明行为的验证、持续维护项目记忆，并用独立 Review 检查声明与生产路径之间的落差。
