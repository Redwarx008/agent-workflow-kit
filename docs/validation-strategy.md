# 验证策略

验证应证明变更的真实因果链，而不是满足统一仪式。

## 确定性行为

默认不得新增或修改测试，除非用户显式要求；已有测试可以作为验证证据运行。用户要求测试时，解析、转换、状态机、资源解析、mesh 不变量、可复现 bug，以及基础设施的鉴权、路径 containment、进程生命周期和格式契约适合自动化回归测试。

## 视觉与 GPU 行为

Shader 编译成功只能证明语法与注册，不证明画面正确。根据问题选择 RenderDoc 截帧、shader 热替换、截图/视觉回归、运行时 smoke test、GPU 状态检查或明确的编辑器人工步骤。

## 编辑器交互

可确定的 service/view-model 行为优先运行现有自动化检查；只有用户显式要求时才新增测试。窗口焦点、native airspace、拖拽、视觉层级等交互使用真实应用验证。不要用纯文本断言代替用户路径。

## 证据门禁

- 每个成功标准必须映射到能证明它的证据。
- 测试名称、类型声明或 selector 存在不等于生产链路已经消费它。
- 原定验证不可运行时，停止并询问用户是否接受替代方案。
- 手动验证尚未执行时明确标注，不得写成已通过。
